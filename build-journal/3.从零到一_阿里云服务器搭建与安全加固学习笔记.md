## 从零到一：我的阿里云服务器搭建与安全加固学习笔记

> 这篇笔记记录了我如何从购买域名和服务器开始，一步步完成安全组配置、SSH密钥登录、备案查询，并最终理解整个建站链路的过程。核心收获是：把“门牌号（域名）”和“房子（服务器）”连起来，远不止设置一条解析那么简单。

---

## 一、初始知识点

> 在学习之前，我只知道“建网站需要买域名和服务器”，但对它们之间如何通信、如何保证安全完全模糊。以下是我已经具备的背景知识，虽然简单，但为后续理解打下了基础。

- **域名是什么**：类似于网站的门牌号，例如 `gyx-a.cn`，方便用户记忆。
- **服务器是什么**：一台放在云上的电脑，有固定的公网IP地址，例如 `118.31.40.119`，用来存放网站文件。
- **解析（DNS）**：将域名指向服务器的IP地址，让用户访问域名时能到达正确的服务器。
- **备案（ICP）**：在中国大陆使用国内服务器必须完成的法定流程，相当于给网站办“身份证”。
- **操作系统**：我的服务器装的是 Ubuntu 24.04，一个常见的 Linux 发行版。
- **远程连接**：通过 SSH 协议登录服务器并执行命令。

一开始我以为只要把域名解析到 IP，网站就能自动打开，现实告诉我，中间还隔着防火墙、密钥、Web服务等多个环节。

---

## 二、学习过程中遇到的问题 & 如何解决的

> 这一路遇到了四个核心问题：看不懂阿里云控制台的各种菜单、搞不清防火墙规则、不理解密钥的作用和格式、不知道如何确认备案状态。每个问题我都经历了从“懵”到“懂”的过程。

### 问题1：阿里云控制台里的“我的资源”都有什么？分别做什么？

**发现过程**  
登录阿里云后，我看到了“概览”页面里列出了一些资源：轻量应用服务器 1 台、域名 1 个、云解析 DNS 1 个、访问控制 RAM 3 个角色。我不知道这些东西各自负责什么，也不知道哪些是必须的。

**我的原始思考**  
我以为“域名”和“云解析”是同一个东西，也以为“RAM”是用来管理服务器内存的。

**学到的原理 / 迭代后的认识**

- **轻量应用服务器**：就是我的那台云电脑（Ubuntu），所有网站程序、数据库都在上面。
- **域名**：我注册的名称 `gyx-a.cn`，只是一个名字。
- **云解析 DNS**：负责把域名翻译成 IP 地址。没有它，域名就是一张废纸。
- **访问控制 RAM**：管理“谁可以操作我的阿里云资源”，比如我可以创建子账号只给查看账单的权限。默认存在的3个角色是系统自带的，不用管。

**解决方案总结**  
对照阿里云控制台的“资源列表”，我逐一搜索了每个产品的文档简介，并用表格记下了它们的作用。最后明确：**服务器 + 域名 + DNS 是建站三件套，RAM 与我当前无关**。

---

### 问题2：安全组（防火墙）怎么配？哪些端口必须开？

**发现过程**  
我在“防火墙”页面看到已有4条规则：TCP 80、TCP 443、TCP 22、ICMP，来源IP都是 `0.0.0.0/0`。我不确定这些是否足够，也不知道 `0.0.0.0/0` 意味着什么。

**我的原始思考**  
我以为端口开得越多越好，甚至想把所有端口都打开，免得以后访问不了。

**学到的原理 / 迭代后的认识**

- `0.0.0.0/0` 表示“允许任何 IPv4 地址访问”，相当于把门完全敞开。
- 端口 `22`（SSH）如果对所有IP开放，会大大增加被暴力破解的风险。
- 端口 `80`（HTTP）和 `443`（HTTPS）必须对所有IP开放，否则用户无法访问网站。
- **最小授权原则**：只开放必要的端口，且尽量限制来源IP。

**解决方案总结**

- 保留 `80` 和 `443` 为 `0.0.0.0/0`。
- 将 `22` 端口改为仅允许我当前的家庭公网IP（`221.176.140.213/32`），但由于我的IP是动态的（重启路由器会变），暂时保持 `0.0.0.0/0`，依靠强密码和后续的密钥登录来保证安全。
- 添加一个临时测试端口 `8080`（用完即删）。

> **认知提升**：安全组不是“越宽松越好”，而是“够用且最小化暴露面”。

### 问题3：SSH密钥是什么？我该怎么生成和使用？

**发现过程**  
阿里云控制台有“密钥对”功能，但我不知道怎么用它。我尝试导入自己生成的公钥时，遇到了错误 `InvalidPublicKeyBody.Malformed`，提示公钥格式不支持。

**我的原始思考**  
我以为是密钥算法太新不被阿里云认可，或者我的生成命令有问题。

**学到的原理 / 迭代后的认识**

- SSH密钥对由 **私钥（自己保管）** 和 **公钥（放在服务器上）** 组成，安全性远高于密码。
- **阿里云控制台导入公钥时，不支持公钥末尾的中文注释**。我当初用 `-C "阿里云服务器密钥"` 生成的 `.pub` 文件末尾有一串中文，导致解析失败。
- 去掉中文注释（只保留 `ssh-ed25519 AAA...` 这一行）后，导入成功。
- 绑定密钥对后，服务器会**自动禁用密码登录**，只能用私钥登录，这是好事。

**解决方案总结**

1. 在本地 PowerShell 用 `ssh-keygen -t ed25519 -f "E:\重要文件\阿里云服务器密钥\aliyun_key" -C "aliyun-key"` 生成密钥对（注释用英文）。
2. 打开 `aliyun_key.pub`，复制内容（确保是一整行，无多余空格）。
3. 在阿里云“密钥对”页面 → 导入已有密钥对 → 粘贴公钥 → 绑定实例 → 重启服务器。
4. 使用 `ssh -i "E:\重要文件\阿里云服务器密钥\aliyun_key" root@118.31.40.119` 连接。

> **小坑**：Windows 对私钥文件权限检查严格，如果报 `Permissions too open`，需要用 `icacls` 命令重置权限。

### 问题4：我怎么知道域名和备案到底成功了没有？

**发现过程**  
域名买了好几天，备案也提交了，但我不确定是否已经生效。访问 `gyx-a.cn` 要么打不开，要么提示未备案。

**我的原始思考**  
我以为备案成功会有短信通知，然后域名就会自动指向我的服务器。

**学到的原理 / 迭代后的认识**

- **域名注册成功 ≠ 解析生效**。需要在云解析DNS中添加 `A` 记录，把域名指向 IP。
- **解析生效 ≠ 网站能访问**。还需要服务器防火墙放行端口、Web服务（如 Nginx）正在运行。
- **备案成功** 需要通过两个渠道确认：
  1. 阿里云ICP备案系统：订单状态变为“已通过”。
  2. 工信部官网（beian.miit.gov.cn）公共查询：输入域名能看到备案号和主体信息。
- 如果工信部查不到，说明备案还在审核中（一般7-20个工作日），或审核不通过。

**解决方案总结**

- **检查解析**：在阿里云域名控制台 → DNS解析 → 确认存在 `@` 和 `www` 的 A 记录指向 `118.31.40.119`。
- **检查备案**：登录阿里云备案系统看状态 → 去工信部官网查询域名 → 两者都显示“通过”才算真正成功。
- **访问测试**：备案通过后，用 `curl http://118.31.40.119` 测试服务器本地响应，再用浏览器访问域名。

> **避坑**：备案期间不要反复修改解析或服务器配置，否则可能被管局判定为“网站内容不稳定”而驳回。

## 三、最终产出

> 经过这一轮折腾，我不仅配置好了服务器，更重要的是建立了一套可复用的建站安全知识框架。以下是这次学习交付的核心认知成果。

1. **我能独立完成阿里云轻量应用服务器的初始配置**，包括：
   - 找到防火墙（安全组）并添加/修改规则。
   - 理解 `0.0.0.0/0` 和 `/32` 的区别。
   - 区分 80、443、22 端口的不同用途。

2. **我能生成并使用 SSH 密钥对登录服务器**，并知道：
   - 如何避免公钥格式错误（注释不用中文）。
   - 如何在 Windows 下用 `ssh -i` 命令连接。
   - 绑定密钥后密码登录会被禁用，这是安全加固的标准做法。

3. **我掌握了域名状态和备案状态的查询方法**：
   - 在阿里云控制台检查域名解析记录。
   - 在阿里云备案系统跟踪订单进度。
   - 在工信部官网核实最终备案结果。

4. **我理解了“域名 → 解析 → 防火墙 → Web服务 → 备案”的完整链路**，不再把任何一个环节视为黑盒。

## 五、个人成长小结

> 这次学习让我从“只会买服务器”到“能独立完成基础安全配置”，最大的跃迁是理解了**安全不是单一设置，而是一条链**。

- **从模糊到清晰**：以前看阿里云控制台就像看天书，现在知道每个菜单是干什么的，知道从哪里找防火墙、从哪里找密钥对。
- **从恐惧到掌控**：之前不敢碰“安全组”和“密钥”，怕配错了服务器连不上。现在知道只要留一个网页版远程连接入口，永远有后路。
- **从零散到系统**：我意识到建站不是“设置一下就行”，而是由 **域名注册 → DNS解析 → 安全组放行 → 服务器Web服务 → 备案** 组成的链条，任何一环断了都不可用。

### 🧭 避坑指南（今日易错点）

- ❌ **不要在公钥注释里写中文**，阿里云导入时会报 `InvalidPublicKeyBody.Malformed`。
- ❌ **不要把所有端口都设为 `0.0.0.0/0`**，尤其是 22 端口，除非你有 fail2ban 或密钥登录。
- ❌ **不要以为备案通过就能立刻访问**，还要检查 Web 服务是否在 80 端口监听。
- ✅ **始终保留一个备用登录方式**：阿里云网页版的“远程连接”不依赖密钥和密码，是救命稻草。
- ✅ **修改任何网络配置前，先拍一个快照**（阿里云“快照”功能），万一崩溃可以回滚。
