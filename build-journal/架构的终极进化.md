# 架构的终极进化：从多仓库解耦到全栈全自动化实战

## 一、 缘起与架构设计哲学 (The Architecture Philosophy)

> “不要把鸡蛋放在同一个篮子里，也不要把所有的代码揉进同一个仓库。”

在搭建这个个人门户之初，我面临着一个痛点：我既有纯原生 HTML 编写的简历，又有海量的算法刷题记录，还有正在连载的 AI 学习笔记。如果把它们全部生硬地塞进一个巨大的博客仓库中，不仅会导致代码极度耦合，还会让本地的 Markdown 写作体验变得支离破碎。

因此，我确立了“去中心化多仓库聚合”的双引擎架构思想。

### 核心目录架构图（通用参考模板）

为了让这套架构具有普适性，我提炼了其底层的物理模型。一套完美的技术门户，应该被拆分为以下三个维度的独立仓库：

Plaintext

```
============================================================
1. 内容生态仓群 (Content Repos) —— [专注纯文本输出]
   如：tech-notes, leetcode-solutions, ai-journey
============================================================
├── README.md
└── 专题分类文件夹/
    ├── index.md                 # 专题导言页 (挂载自动化卡片组件)
    ├── 第一篇文章.md             # 纯粹的技术沉淀
    └── 第一篇文章.assets/        # (核心规范) 专属的图片存放路径

============================================================
2. 基础设施仓 (Infrastructure Repo) —— [专注静态门面与运维]
   如：resume-project
============================================================
├── my-resume/                   # 纯 HTML/Tailwind CSS 编写的原生简历
│   └── index.html
└── docs-md/                     # 博客建站日志等工程记录

============================================================
3. 展现层主引擎仓 (Engine Repo) —— [专注框架渲染与 CI/CD]
   如：my-blog (VitePress 核心)
============================================================
├── .github/workflows/
│   └── deploy.yml               # 掌控全局的自动化流水线图纸
├── package.json
└── docs/
    ├── index.md                 # 网站首页大门
    ├── .vitepress/
    │   ├── config.mjs           # 全局路由与配置中枢
    │   └── components/
    │       ├── BentoGrid.vue    # 首页炫酷便当盒 UI
    │       └── CategoryList.vue # 数据驱动的自动化分类卡片
    ├── public/
    │   └── resume/              # 🔗 指向基础设施仓简历的软链接 (原样直通)
    │
    ├── tech-notes/              # 🔗 指向内容仓 1 的软链接 (VitePress 编译)
    └── leetcode-solutions/      # 🔗 指向内容仓 2 的软链接 (VitePress 编译)
```

**设计精髓：** 各个内容仓库在物理层面上完全隔离，但在主引擎的 `docs` 目录下，通过操作系统的 `mklink /J`（软链接/目录联接）被缝合在一起。对 VitePress 来说，它们就是一个完整的网站；对底层的 Git 来说，它们依然是相互独立的项目。

## 二、 工程化破冰：踩穿环境与路由的重重深坑

> 工程落地的过程往往伴随着各种意外的“坑”，但正是解决这些架构冲突和环境报错的过程，重塑了我的全栈工程思维。

### 1. Node.js 环境的权限与路径迷失 (`EPERM`)

在初始化博客时，执行 `npm add -D vitepress` 爆出 `Error: EPERM, mkdir... cache`。

- **复盘认知：** 这暴露了我对 Windows 终端权限的盲区。由于 Node.js 装在非系统盘，普通终端无权跨盘创建缓存。
- **解决：** 必须使用**管理员身份**运行 CMD，且跨盘跳转时不能只用 `cd`，必须先敲盘符（如 `E:`）切入工作盘，再执行环境构建。

### 2. VitePress 启动崩溃：动态路由与文件夹规范陷阱

运行本地服务时，系统报出 `Missing paths file for dynamic route` 和 `ENOTDIR`。

- **复盘认知：**
  1. 为了好看，我在 Markdown 文件名里加了中括号 `[]`，这在 Vite 生态里被误认为了“动态路由参数”，导致引擎解析错乱。
  2. 我在初始化时错误地将 `.vitepress` 建立成了一个文本文件，而它本该是一个隐藏文件夹。
- **解决：** 永远不要在工程文件名中使用空格、中文全角括号和 `[]`，统一使用“小写英文/数字+中划线”；删掉错误文件，重建规范的大脑控制夹。

### 3. 数据隔离：纯 HTML 简历如何不被博客主题污染？

- **复盘认知：** 如果把原生 HTML 简历当作普通文章放入 VitePress，它的排版会被强制套上博客主题而崩溃。
- **解决：** 利用现代 SSG 框架的静态资源直通机制。我将简历软链接到了 `docs/public/resume/` 目录下。在这里面的文件编译时会被原封不动地直通网站根目录，实现了“统一域名，内核隔离”。

## 三、 前端架构进化：数据驱动视图与渐进式体验

> 解决了底层环境问题，接下来的挑战是如何让信息的呈现更加优雅且“零维护”。

### 1. 混合驱动路由：Vue 组件消灭“复制粘贴地狱”

随着文章增多，每次都要在 `config.mjs` 和专栏的 `index.md` 里手写两遍路由，简直是灾难。

- **架构升级：** 我手写了一个全局 Vue 组件 `<CategoryList/>`。利用 VitePress 的 `useData()` API，让组件自动捕获当前 URL 路径，反向去读取 `config.mjs` 中的 `sidebar` 字典，并动态渲染成现代化的 CSS Grid 卡片阵列。
- **结果：** 从此各专栏大门只需一行 `<CategoryList/>`。实现了“核心路线在 `config.mjs` 手动掌控，海量长尾文章通过组件自动渲染”的混合驱动模式。

### 2. 漏斗式 UX 设计：Bento Box 布局

针对 HR、技术面试官和同行不同的阅读耐心，我将网站设计为渐进式展现 (Progressive Disclosure)：

- 首页采用纯 CSS 的 **Bento Box (便当盒)** 布局，直接打出 GPA 3.74、Top 5% 及中级软件设计师认证等硬核指标。
- 内页利用 VitePress 的 "On this page" 树状导航配合滚动高亮，并开启 Local Search，保障硬核笔记的沉浸式阅读与检索体验。

## 四、 云原生实战：跨越生产环境的死亡壁垒

> 本地预览的完美并不等于生产环境的成功。从本地到阿里云服务器的部署路上，我跨越了四个 DevOps 深坑。

### 1. 本地图片的云端 404 惨案

- **发现：** 网站上线后图片全变乱码，因为 Typora 插入的是 `E:\Web_git\...` 绝对路径。
- **解决：** 云端 Linux 根本没有 E 盘。必须在 Typora 中强制设置图片保存路径为 `./${filename}.assets`，并优先使用相对路径，从根源掐断物理路径泄露。

### 2. 跨平台依赖的二进制冲突 (Exit Code 127)

- **发现：** 流水线执行 `vitepress build` 时报 `Permission denied`，编译中断。
- **解决：** 深刻理解了 Git “传图纸不传砖块”的哲学。通过严格配置 `.gitignore` 拦截 Windows 本地的 `node_modules`，让云端 Linux 环境自己拿着 `package.json` 下载原生依赖。

### 3. Nginx 安全组与 SNI 拦截

- **发现：** 配置了 HTTPS 后，用 IP 直连被浏览器报“不安全”拦截；随后流水线 SCP 上传时又遭遇 `i/o timeout`。
- **解决：** 前者是 SNI 匹配失败的安全机制（IP 对不上证书域名）；后者是因为我早期的阿里云安全组物理锁死了 22 端口白名单。放宽 IP 后，结合极高安全级别的 SSH 密钥顺利打通隧道。

### 4. Git 底层认知跃迁：软链接导致的跨库失效

- **发现：** 修改了内容仓库的代码，本地 `my-blog` 软链接实时更新，但我却无法 Push 主仓库触发 GitHub Actions 部署。
- **解决：** 彻底看透了 Git 对软链接的追踪逻辑——Git 追踪的是软链接的“指向地址”**，而不是其内部的“内容”。既然地址没变，主仓库就认为自己是 0 字节修改。因此，我确立了**空提交 (Dummy Commit) 和 **Workflow Dispatch** 两种点火机制来手动唤醒云端工厂。

## 五、 面试高频考点总结

将这套主页背后的思考逻辑带入面试，是展现全栈工程素养的最佳利器。

| **面试考点 / 场景**       | **对应的本次实践知识点**                                     |
| ------------------------- | ------------------------------------------------------------ |
| **持续集成与云原生部署**  | GitHub Actions 工作流机制；解决跨平台 Exit Code 127 及 SSH 免密 SCP 传输。 |
| **前端组件与数据驱动**    | 如何编写 `<CategoryList/>` Vue 组件提取 `useData()`，实现数据驱动 UI。 |
| **Git 底层文件追踪机制**  | 解释修改软链接指向的内容时，宿主仓库不产生 Commit 的原因（文件指针与内容分离）。 |
| **操作系统底层与隔离**    | 软链接（Directory Junction）的原理；利用 `public` 保护区底线隔离原生 HTML 防止框架污染。 |
| **网络安全与 Nginx 网关** | Nginx 的 `try_files` 兜底路由陷阱；SNI 匹配失败原理及 HTTPS (Let's Encrypt) 的配置逻辑。 |

## 🚨 附录：个人主页日常维护与自动化更新标准操作手册 (SOP v2.0)

> “架构的优劣，取决于日常维护的阻力大小。” 既然我已经把复杂的脏活累活都交给了云端的 GitHub Actions，那么在本地，我只需要像操作一台精密的仪器一样，遵守这套标准化的指令流。

### ⚠️ 核心认知预警：谁来扣动“自动部署”的扳机？

我的自动化流水线图纸 (`deploy.yml`) 存放在主引擎仓库 `my-blog` 里。

这意味着：**如果我仅仅在内容仓库（如 `tech-note`）推送了代码，云端工厂是不会开工的。** 每一次更新的最后一步，都必须是“向主引擎发出版控指令”，从而触发整条流水线的运作。

### 场景一：对已有博客进行只字不改的“纯文本修改”

这是最简单的维护场景（如改错别字、补充感悟），不涉及任何新路径的产生。

- **步骤 1：本地修改代码**

  打开本地对应的真实源文件夹（例如 `E:\Web_git\tech-note\`），用 Typora 修改 `.md` 文件并保存。

- **步骤 2：提交内容仓库**

  打开 GitHub Desktop，切换到 `tech-note` 仓库。填写提交信息（如：`fix: 修改错别字`），点击 Commit 并 Push origin。

- **步骤 3：触发引擎重构（极其重要）**

  切换到主引擎仓库 `my-blog`。为了触发流水线，我可以在 `README.md` 里敲一个空格或写句更新日志，执行 **空提交 (Dummy Commit)** 然后 Push。

  *(💡 进阶：直接去 GitHub 网页端 `my-blog` 的 Actions 界面，点击 `Run workflow` 按钮手动触发，连假装提交都省了。)*

### 场景二：在已有的知识库中“新建内容”

得益于之前开发的 `<CategoryList/>` 组件，我不需要去修改专栏的导言页，只需按规范更新全局路由字典。

- **步骤 1：本地创作与资源隔离**

  - 在目标仓库新建 `.md`。使用 Typora 时，图片保存在自动生成的 `./assets/` 相对路径中。**不需要把 `.assets` 写入任何配置文件，框架会自动识别。**
  - *若是纯代码文件 (`.py`)*：VitePress 侧边栏不支持直接挂载代码。必须新建 `.md`，使用 `<<< @/专栏目录/你的代码.py` 语法将代码完美嵌入并高亮。

- **步骤 2：更新 VitePress 路由字典**

  打开主引擎配置：`docs\.vitepress\config.mjs`。找到对应专栏的 `sidebar` 数组，新增一行登记新 Markdown 的路径（注意去掉 `.md` 后缀）。

- **步骤 3：双轨推送（一次性触发）**

  先在内容仓库 Push 新文章。接着切换到 `my-blog`，把刚才修改过路由的 `config.mjs` Push 上去。云端流水线会自动拉取新文章，各专栏导言页也会自动渲染出新卡片。

### 场景三：开辟全新领域，将“新仓库”接入个人主页

随着技能树扩展，如果新建了完全独立的仓库（例如 `java-backend-notes`），需进行一次架构级拼装。

- **步骤 1：建立专栏大门 (导言页)**

  在新仓库的根目录下，新建 `index.md`，写入专栏介绍，并在末尾加上 `<CategoryList/>` 组件。正常 Push（千万别忽略它）。

- **步骤 2：创建本地虚拟映射 (用于本地预览)**

  以管理员身份运行 CMD，敲入指令：`mklink /D "E:\Web_git\my-blog\docs\java-notes" "E:\Web_git\java-backend-notes"`

- **步骤 3：切断 Git 误判 (极度重要)**

  打开 `my-blog/.gitignore` 文件，加上一行 `docs/java-notes/`，防止主仓库把这些虚拟文件误传上云端。

- **步骤 4：更新云端拼装图纸 (`deploy.yml`)**

  在“微服务模块拉取”中，复制并添加新的 Checkout 步骤，Repository 指向新仓库。

- **步骤 5：配置全局导航 (`config.mjs`)**

  在 `nav` 数组中增加顶部按钮，在 `sidebar` 中为新专栏配置目录分组逻辑。

- **步骤 6：全量推送上线**

  确保新仓库已 Push。切换回 `my-blog`，将变动的 `.gitignore`、`deploy.yml` 和 `config.mjs` 一次性 Push 上云，触发全自动架构重组。

### 场景四：架构级代码调整与页面修改边界 (The Decision Matrix)

当进行微调或优化时，严格遵循“物理源头修改法则”。绝对禁止在主仓库的 `docs/public/` 等编译目标目录中修改任何文件。

| **我的修改意图**           | **我必须去哪里修改源文件？**            | **底层逻辑说明**                               |
| -------------------------- | --------------------------------------- | ---------------------------------------------- |
| **调整博客文章的具体内容** | `tech-note` 等对应的**内容仓库**。      | 业务内容的唯一物理源头，供流水线随时拉取。     |
| **修改专栏的前言文案**     | 各个内容仓库根目录下的 `index.md`。     | 导言与 `<CategoryList/>` 组件随内容源流转。    |
| **修改全局导航栏、侧边栏** | `my-blog/docs/.vitepress/config.mjs`    | 这是主引擎的全局路由字典与控制大脑。           |
| **更改自动化部署逻辑**     | `my-blog/.github/workflows/deploy.yml`  | 定义云端服务器“怎么干活”的 CI/CD 蓝图。        |
| **修改纯 HTML 简历内容**   | `Individual-Resume` 仓库的 `index.html` | 必须在原生仓库修改，防止被云端流水线潮水覆盖。 |

**线上回归验证 (Post-Deploy Audit)：**

部署绿灯后，打开网站，使用 `Ctrl + F5` 强制刷新，或开启 `Disable cache`，防止浏览器加载旧的 CSS 缓存导致“修改未生效”的假象。
