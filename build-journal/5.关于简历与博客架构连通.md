  ## 一、初始知识点

  > 凡事预则立，在动手搭建这套现代化的个人主页之前，我已经具备了一些基础的工程化能力和写作习惯，这为后续的架构整合打下了很好的底子。

  在本次实践之前，我已经掌握了以下技能和知识储备：

  2. **版本控制与本地工作流：** 我习惯使用 Git 配合 GitHub Desktop 进行代码和文档的版本管理，所有的学习仓库（如 AI 学习路径、力扣题解、阅读心得等）都已经分类存放在本地电脑的 `E:\Web_git\` 目录下，并与远端 GitHub 保持同步。
  3. **Markdown 写作习惯：** 我平时习惯使用 Typora 进行沉浸式的技术文档编写，并且摸索出了一套本地 `assets` 文件夹存放图片的管理方式，保证了文档在本地的数据安全与独立性。

  ## 二、学习过程中遇到的问题 & 如何解决的

  > 工程落地的过程往往伴随着各种意外的“坑”，但正是解决这些架构冲突和环境报错的过程，重塑了我的全栈工程思维。

  ### 问题 1：如何让分散的本地仓库聚合成一个博客，且不破坏原有的 GitHub 同步？

  - **发现过程：** 我的力扣题解、AI 学习记录等内容分散在不同的本地文件夹里（且各自独立绑定了 GitHub 仓库）。如果直接把它们移动到 VitePress 的 `docs` 目录下，会导致原有的 GitHub Desktop 同步路径全部失效，全部需要重新拉取和配置。

  - **我的原始思考：** 我一开始以为只能放弃原有的独立仓库，把所有文件强行复制合并到一个新的巨大的“博客仓库”中，或者必须忍痛在本地重新搞一套复杂的 Git Submodule 嵌套结构。

  - **学到的原理 / 迭代后的认识：** 我学到了操作系统底层的一个高级魔法——**软链接（目录联接 Directory Junction）**。它可以在不移动真实物理文件的前提下，在一个新目录中创建一个指向源文件夹的“任意门”。对于代码编辑器和打包工具来说，文件就像真的在这个目录里一样；而对于底层的 Git 来说，物理文件依然在原地，互不干扰。

  - **解决方案总结：**

    > 流程梳理：
    >
    > 1. 新建 `my-blog/docs` 空外壳目录。
    > 2. 使用管理员权限打开 CMD 终端。
    > 3. 利用 `mklink /J "目标虚拟路径" "本地真实源路径"` 命令，把原来分散的 5 个仓库映射到 `docs` 下的各个子目录中。这样既保全了旧的 Git 提交习惯，又让 VitePress 能瞬间读取到所有文章。

  ### 问题 2：纯 HTML 的简历如何与 Markdown 编译的博客和谐共存？

  - **发现过程：** 我的简历 `Individual-Resume` 是纯原生 HTML 加上 Tailwind CSS 写的。如果把它也作为一篇文章塞进 VitePress，VitePress 可能会强行给它套上博客的主题样式，导致简历排版彻底崩溃。
  - **我的原始思考：** 我曾想过是不是要把简历单独部署一个域名，博客再部署另一个域名，然后两边互相放链接跳转。但这在维护上非常割裂。
  - **学到的原理 / 认识：** 现代静态站点生成器（SSG）都有静态资源直通（Public 目录）的机制。在 VitePress 中，所有放在 `public` 文件夹里的内容，在编译打包时都不会被转化，而是原封不动地搬运到网站根目录。
  - **解决方案总结：** 我利用 `mklink` 将简历仓库映射到了 `docs/public/resume/` 目录下。这样一来，简历就成了博客网站下的一个独立子页面（`域名/resume/index.html`）。我在 HTML 简历中加上跳转按钮，面试官点击后就能丝滑地进入由 VitePress 渲染的博客内页，实现了“门面”与“内核”的完美结合。

  ### 问题 3：Node.js 环境配置时的 EPERM 权限报错与路径迷失

  - **发现过程：** 在终端初始化博客时，先是遇到了 `无法将“npm”项识别` 的报错；装好 Node.js 后，执行 `npm add -D vitepress` 时又爆出了一大串红字 `Error: EPERM, mkdir... cache`。

  - **学到的原理 / 迭代后的认识：** 第一，`npm` 命令失效是因为系统环境变量（PATH）里没有 Node.js 的路径。

    第二，`EPERM` 报错是因为我把 Node 装在了非系统盘（D盘），而普通的终端没有在根目录跨盘创建缓存文件夹的管理员权限。此外，Windows 的 CMD 在跨盘符跳转时，必须先敲盘符（比如 `E:`），仅仅用 `cd` 是切不过去的。

  - **解决方案总结：** 关闭报错的终端，以**管理员身份**重新运行 CMD，先输入 `E:` 切换盘符，再 `cd` 进入博客目录。有了最高权限且路径正确后，npm 顺利下载了所有的依赖包。

  ### 问题 4：VitePress 启动时的 Dynamic Route 与 ENOTDIR 缓存报错

  - **发现过程：** 当我激动地敲下 `npm run docs:dev` 准备看效果时，系统提示 `Missing paths file for dynamic route`，接着又报 `ENOTDIR: not a directory, mkdir cache`，导致页面无法生成。

  - **我的原始思考：** 我以为是我 `config.mjs` 里的代码写错了，或者哪里的路由配置少写了括号。我甚至在资源管理器里对着 `.vitepress` 看了半天没觉得哪里不对。

  - **学到的原理 / 迭代后的认识：**

    这其实暴露出两个严重的文件规范意识问题：

    1. **路由命名规范：** 我为了好看，在力扣的 Markdown 文件名里加了中括号 `[]`。但在 Vite 生态里，中括号代表“动态路由参数”（类似于占位符），导致引擎解析错乱。
    2. **工程目录规范：** 我在 Trae 里新建 `.vitepress` 时，错误地把它建成了一个**文本文件**（File），而它本该是一个**隐藏文件夹**（Folder）。系统想要在这个“文件”里面新建 `cache` 目录，自然会提示“这不是一个目录”。

  - **解决方案总结：**

    > 流程梳理：
    >
    > 1. 将旧仓库中所有带中括号 `[]` 和特殊符号的 Markdown 文件名，全部重命名为规范的英文加短横线格式（如 `01-two-sum.md`），把漂亮的原标题放进文件内部的一级标题里。
    > 2. 删掉错误的 `.vitepress` 文本文件，重新创建一个同名的**文件夹**。
    > 3. 在文件夹内新建 `config.mjs` 并写入配置代码，问题瞬间迎刃而解。

  ### 问题 5：海量文章手动配置太痛苦，能否让网站“自动扫描”文件夹生成菜单？

  - **发现过程：** 搞懂 `config.mjs` 的控制逻辑后，我发现每次在力扣加一道题，都要去配置文件里加一行，如果以后有上百道题，手动配置简直是个灾难。我希望用数字标号在本地排序，让网站自动去扫盘。

  - **我的原始思考：** 为什么 GitHub 或者 VitePress 不能自动识别文件夹去动态补充代码呢？直接做一个“免维护”的侧边栏不好吗？

  - **学到的原理 / 迭代后的认识：** 这里面藏着静态网站生成器（SSG）的核心哲学。

    1. **静态托管无后台：** 网站上线到 GitHub Pages 后是纯静态的，用户访问时 GitHub **绝对不会**在后台运行代码去帮你扫盘。
    2. **原生的克制：** VitePress 追求极致性能与知识树的“精准修剪”，为了防止把 `README.md` 或草稿也扫出来，官方默认只支持手动登记。
    3. **混合驱动模式（Hybrid Routing）：** 虽然运行期不能扫，但在本地打包的“编译期”是可以扫的。因为 `config.mjs` 本质上是一个可以在编译时执行的 Node.js 脚本！

  - **解决方案总结：** 我明确了“高价值手动掌控文案，海量长尾交给自动化”的混合管理策略。

    对于“项目实战”、“AI探索”这种文章少但价值高的专栏，继续在 `config.mjs` 里硬编码手动配置，以保证网页展示的标题足够惊艳、层级足够灵活。

    对于未来可能产生海量题目的“力扣题解”，我可以在 `config.mjs` 顶部引入 Node.js 的 `fs`（文件系统）模块自己写一段扫描脚本，或者直接引入社区成熟的 `vitepress-sidebar` 编译期插件。它会在我每次运行打包时自动把带有数字标号的 md 文件扫出来编成目录，彻底解放双手。

  ### 问题 6：后续如何高效、规范地添加新的文件夹与 Markdown 文档？

  - **发现过程：** 随着我学习和刷题的推进，我必然会面临两种高频的长尾场景：一是在现有的专栏里写一篇新文章（比如多做了一道力扣题），二是开辟一个全新的技术大分类（比如新学了某个技术栈，本地新克隆了一个独立仓库）。我需要一套标准的、机械化的标准操作流，确保以后添加内容时不会乱套。

  - **学到的原理 / 迭代后的认识：** 我彻底理清了“物理存储（硬盘）— 软链接隧道（操作系统）— 路由门面（VitePress 配置）”三者之间的层级协作关系。物理文件和软链接只决定文件在系统打包时“存不存在”，而 `config.mjs` 决定了网页上“能不能让用户看到”。

    基于这个原理，后续的扩展操作可以完美地固化为两套标准流，不需要反复试错。
  
  - **解决方案总结：**
  
    我将后续的迭代流程提炼为以下两类标准的工程执行手册：
  
    > **场景一：在已有的专栏（如 leetcode 或 resume-project）中添加新文章**
    >
    > 1. **内容生产：** 直接用 Typora 打开本地对应的物理文件夹（例如 `E:\Web_git\Individual-Resume\docs-md\`），新建 `.md` 文件并撰写内容。牢记文件名必须严格使用“小写英文/数字+中划线”的无特殊字符规范（如 `03-xxx.md`）。
    > 2. **界面登记：** 打开 `my-blog/docs/.vitepress/config.mjs`，在对应专栏的 `sidebar` 节点下的 `items` 数组中，手动追加一行该新文章的展示标题与相对路径，保存后本地即可即时预览。
  
    > **场景二：新开辟一个全新的技术大分类（如本地新增了独立的 `vue-notes` 仓库）**
    >
    > 1. **物理打通：** 以管理员身份运行 CMD 终端，执行 `mklink /J` 命令，在 `my-blog/docs/` 下为这个新仓库创建一个软链接“任意门”。
    >
    > > 比如：mklink /J "E:\Web_git\my-blog\docs\resume-project" "E:\Web_git\Individual-Resume\docs-md"
    >
    > 1. **框架控制：** 打开 `config.mjs` 配置文件进行两处框架层面的微调：
    >    - 在 `nav` 数组中，为顶部全局导航栏新增一个直达该专栏的按钮。
    >    - 在 `sidebar` 对象中，为该新路径（如 `/vue-notes/`）全新声明一套左侧目录树，并挂载该分类下的首批初始文章。

  **🚨 附：本地环境终极避坑指南**
  
  - **路由禁区：** 永远不要在工程文件名或路径中使用空格、中文字符、特殊标点和中括号 `[]`，统一使用“小写英文/数字+中划线”！需要漂亮标题直接写在 md 文件的第一行（H1）。
  - **控制中心定义：** `.vitepress` **必须是一个文件夹**，它是整个网站的大脑，绝对不能建成了文本文件！
  - **权限管理意识：** 凡是涉及到全局包安装、软链接映射，记得顺手切出“管理员身份运行 CMD”。
  - **CMD 跨盘迷思：** Windows 的终端里，`cd E:\` 没用，必须先敲 `E:` 才能切进你的工作盘，这一个回车能帮你省去一堆莫名其妙的系统盘污染问题。
