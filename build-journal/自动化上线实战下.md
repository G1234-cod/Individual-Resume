**自动化上线实战（下）：跨越云原生架构与多仓库 CI/CD 部署的深坑**

## 一、初始知识点

> 真正的自动化不仅是写好代码，更是让代码能在云端独立运转。在这次实战前，我已经打下了一些基础，这成为了我排错时的底层逻辑支撑。

在开始构建这套基于 GitHub Actions 的 CI/CD 流水线之前，我已经掌握了以下技能储备：

1. **服务器基础与防御直觉：** 我已经在阿里云 Ubuntu 服务器上成功配置了 UFW 防火墙，并懂得使用更高级别的“SSH 密钥 + Passphrase 密码短语”进行安全登录，还在云控制台配置了安全组。
2. **多仓库隔离与本地聚合：** 我将原生 HTML 简历、技术笔记、项目记录等分成了独立的 GitHub 仓库，并通过 Windows 的软链接（`mklink`）在本地聚合成了一个完整的 VitePress 博客以供实时预览。
3. **Nginx 核心网关认知：** 我已经理解了 Nginx 作为 Web 服务器的定位，并掌握了其监听 80/443 端口的基础配置逻辑。

## 二、学习过程中遇到的问题 & 如何解决的

> 从单机思维跨越到云原生 DevOps 架构，中间隔着无数个由环境差异引发的坑。每一次排错，都是一次对计算机底层原理的重塑。

### 问题 1：Nginx 启动死锁与 SSL 证书的“先有鸡还是先有蛋”

- **发现过程：** 在初始化 Nginx 时，我直接写入了包含 HTTPS 和证书路径的配置文件，结果 Nginx 报错 `cannot load certificate... No such file`，服务彻底挂掉。

- **我的原始思考：** 我本想一步到位把安全配置写好，却忽略了此时我还没有用 `acme.sh` 申请出证书。没有证书 Nginx 起不来，Nginx 起不来又无法通过域名的 HTTP 验证去申请证书。

- **学到的原理 / 迭代后的认识：** Nginx 在启动时会严格校验配置文件里的每一个路径。必须为证书申请提供一个“干净的温床”。

- **解决方案总结：** > 破局步骤：

  > 1. 先用 `.bak` 备份写好的 HTTPS 配置。
  > 2. 编写一个临时的纯 80 端口 HTTP 配置，专门放行 `/.well-known/acme-challenge/` 路径供 Let's Encrypt 验证。
  > 3. 使用 `acme.sh` 成功申请并挂载证书后，再把原来的 HTTPS 配置文件还原回来，最后重载 Nginx。

### 问题 2：用 IP 访问网站被浏览器拦截提示“不安全” (SNI 匹配失败)

- **发现过程：** 证书配好后，因为域名还没备案，我尝试用 `http://118.31.40.119` 直接访问，结果浏览器报红提示“您的连接不是私密连接”。
- **我的原始思考：** 难道是我刚申请的 Let's Encrypt 证书坏了，或者 Nginx 配置写错了？
- **学到的原理 / 迭代后的认识：** 这是标准的 **SNI (Server Name Indication) 匹配失败** 安全机制。我的证书是颁发给 `gyx-a.cn` 的，当浏览器用 HTTPS 访问纯 IP 时，发现访问地址和服务器掏出的“身份证（域名）”对不上，为了防止中间人攻击就会拦截。
- **解决方案总结：** 这不是故障，点击浏览器的“继续前往不安全页面”即可预览。如果想完美解决，只需在 Nginx 中单开一个 80 端口的 `server` 块专门处理 IP 直连（不强制跳转 HTTPS）即可。

### 问题 3：本地软链接在云端失效与多仓库聚合打包难题

- **发现过程：** 我本想把配置好的主引擎仓库（`my-blog`）推送到 GitHub，以为云端能像我本地电脑一样，顺着软链接把其它仓库的文章一起打包。结果发现云端打包出来的全是一堆无法解析的空壳。

- **学到的原理 / 迭代后的认识：** GitHub 和云服务器是完全“瞎”的，它们只认 `.git` 根目录下的物理文件，根本不认 Windows 的带蓝色箭头的软链接（快捷方式）。

- **解决方案总结：** 引入 **“云端流水线组装 (Cloud Assembly)”** 架构。

  > 架构重构流程：
  >
  > 1. 本地保持软链接用于实时预览，但在主引擎仓库配置 `.gitignore` 将它们全部拦截，不推上云。
  > 2. 在 `.github/workflows/deploy.yml` 脚本中，利用多个 `actions/checkout@v4` 步骤，让云端服务器在编译前主动去各个独立的子仓库（如 `tech-note`）拉取源码，精准拼装进云端的 `docs` 目录。
  > 3. 拼装完毕后，再统一执行 `npm run docs:build`。

### 问题 4：Windows 隐藏扩展名陷阱导致忽略文件失效

- **发现过程：** 我明明写了 `.gitignore`，但配置流水线时发现它并没有起作用，海量的本地文件依然被推了上去。
- **学到的原理 / 迭代后的认识：** Windows 系统默认隐藏已知文件扩展名。我新建的文本文件其实叫 `.gitignore.txt`，而 Git 门卫极其严格，只认 `.gitignore` 这一个名字，多一个后缀都不行。
- **解决方案总结：** 在文件资源管理器勾选“显示文件扩展名”，删掉 `.txt` 后缀，系统弹出警告点击“是”，成功激活拦截规则。

### 问题 5：Windows 与 Linux 的跨平台二进制冲突 (Exit Code 127)

- **发现过程：** 流水线走到 VitePress 编译那一步时，突然崩溃，报错 `sh: 1: vitepress: Permission denied`，并抛出 `Process completed with exit code 127`。
- **学到的原理 / 迭代后的认识：** 这是 DevOps 中最经典的“跨界水土不服”。因为前面 `.gitignore` 没配好，我把本地几百兆的 Windows 版 `node_modules` 推上了 GitHub。当云端 Linux 服务器尝试运行这些为 Windows 编译的二进制执行文件时，自然被系统底层的权限拒绝了。**Git 的核心哲学是：只传“图纸（package.json）”，不传“砖块（node_modules）”。**
- **解决方案总结：** 在本地物理删除 `node_modules` 并提交推送，给云端仓库“清创”。让流水线里的 `npm install` 步骤拿着图纸，自己在 Linux 环境下下载最适配的原生依赖。

### 问题 6：本地绝对路径导致云端编译崩溃 (Rollup failed to resolve import)

- **发现过程：** 依赖修好后，流水线终于开始打包，但中途再次报错 `failed to resolve import "/Web_git/.../image.png"` 并中断。
- **学到的原理 / 迭代后的认识：** Markdown 写作的典型灾难。我在本地 Typora 插入图片时生成了带盘符的绝对路径（如 `E:\Web_git`）。这在本地没问题，但云端 Linux 根本没有 E 盘。这暴露了本地环境与生产环境物理隔离的脆弱性。
- **解决方案总结：** 全局搜索所有的 Markdown 仓库，把绝对路径全部改为相对路径（如 `./xxx.assets/image.png`）。同时，在 Typora 偏好设置中强制勾选“优先使用相对路径”，从根源切断此隐患。

### 问题 7：安全组物理阻断导致 SCP 部署超时

- **发现过程：** 流水线走到了最后一步 `Deploy to Aliyun Server`，眼看就要胜利，却报出 `dial tcp ***:22: i/o timeout` 网络连接超时。
- **我的原始思考：** 难道是我的 SSH 密钥（Secrets）填错了？
- **学到的原理 / 迭代后的认识：** 密钥没填错，是我前期的安全防护太硬核了。我曾在阿里云控制台把 22 端口（SSH）的白名单锁死成了我自己的公网 IP。导致身在海外的 GitHub 服务器敲门时，直接被云安全组的底层物理防火墙无视了。
- **解决方案总结：** 因为我使用了极高安全级别的“SSH 密钥 + 密码短语”双重认证，无需担心暴力破解。前往阿里云控制台将 22 端口的来源 IP 放宽至 `0.0.0.0/0`，在 GitHub 页面点击 `Re-run jobs`，瞬间部署成功。

### 问题 8：Nginx 兜底路由与 rsync 漏传造成的网站“假象”

- **发现过程：** 网页上线后，原生的 HTML 简历失去了样式（变成黑底蓝字），且部分博客点进去报 404 的地方，全显示一句丑陋的 `Hello from gyx-a.cn`。不仅如此，个别文章也因为名字不对找不到。

- **学到的原理 / 迭代后的认识：**

  1. 那句 Hello World 是我当初测试 Nginx 时残留的文件，配合着 Nginx SPA 专属的 `try_files $uri /index.html` 兜底路由，导致它强行接管了所有的 404 页面。
  2. 样式丢失是因为我在云端 YAML 脚本里提取简历时，只用命令拷贝了外层文件，漏掉了关键的 `css/assets` 资源文件夹。
  3. 文章找不到是因为我在本地更改了软链接里的文件名，却没有将真正底层仓库（`Individual-Resume`）里的原名同步提交推送到云端。

- **解决方案总结：**

  > 全方位修补方案：
  >
  > 1. 登录服务器 `rm` 删除旧的 `index.html`，并将 Nginx 的兜底路由改回严格的 `try_files $uri $uri/ =404;`。
  > 2. 升级 YAML 脚本，使用强大的 `rsync -av --exclude='.git'` 命令完美同步原生简历下的所有附属资源文件夹。
  > 3. 在本地找到底层仓库真实文件，规范重命名并 Push 同步到云端。

### 问题 9：混合架构下的页面排版统一与代码修改边界迷失

- **发现过程：** 在网站搭建成型后，我试图修改页面的排版。我一开始想让 HTML 简历也带上 VitePress 的侧边栏，并且在修改代码时，困惑于到底该改 `config.mjs`，还是直接去改 `my-blog/docs/public/resume` 目录下的 HTML 文件。
- **我的原始思考：** 我以为既然网站是一个整体，那就应该在一个统一的配置文件里控制所有页面的导航栏和排版；并且我以为既然最终生成的文件在 `public/` 下，直接改那里最快。
- **学到的原理 / 迭代后的认识：** 我深刻理解了 **SSG（静态站点生成）与传统 Web 架构的“视图解耦”原理**，以及流水线架构下的“防覆盖”铁律。
  1. **双引擎的权力隔离：** VitePress 负责的是“知识库”，`config.mjs` 是控制中心，它在编译期为所有的 Markdown 笔记统一生成头部和侧边栏。而 HTML 简历是作为“外来独立文件”放入 `public/` 保护区的，它完全脱离 VitePress 的控制，也就是“不需要、也不该有”统一的侧边栏。
  2. **潮水与沙滩理论：** 绝对不能直接修改流水线的目标编译文件夹（如 `my-blog/docs/public/resume`）。因为这里是“沙滩”，下次 GitHub Actions 一跑，源仓库拉取的新文件就像“潮水”一样，会把我在这里直接修改的代码全部无情覆盖。
- **解决方案总结：** 彻底放弃“用一个配置管所有页面”的执念，实行“分而治之”。建立严格的代码修改决策矩阵：要改全局框架去主引擎改 `config.mjs`，要改个性化排版去底层源仓库改 HTML。

## 五、个人成长小结

> “服务器是 Cattle（牲口），代码和流水线才是 Pets（宠物）。”这是我认知上最大的跃迁。

经过这次实战，我彻底打破了“配置环境是一次性玄学”的恐惧。我认识到，只要 GitHub 里的图纸（`package.json`）和自动化图纸（YAML）正确无误，就算现在把阿里云服务器格式化，机器也能在几分钟内自动帮我一比一重建整个庞大的知识库。

我也深刻体会到了保持工程环境纯粹性的重要性。不能把本地的“建筑废料”（带后缀的文本、Windows 的依赖包、绝对盘符路径）带上云端。云端的事情，就要用云原生的标准去解决。

这是一份为你量身定制的、可直接作为笔记结尾的**终极版标准操作手册 (SOP)**。我已经将所有的日常维护场景（包括你提到的三种情况，并补充了底层逻辑）全部结构化，你可以直接将以下内容复制并替换掉上一版笔记末尾的附件部分。

## 🚨 附：个人主页日常维护与自动化更新标准操作手册 (SOP)

> “架构的优劣，取决于日常维护的阻力大小。” 既然我已经把复杂的脏活累活都交给了云端的 GitHub Actions，那么在本地，我只需要像操作一台精密的仪器一样，遵守一套标准化的指令流。

为了保证多仓库聚合架构的稳定运转，我总结了以下三种最高频的日常运作场景。

### 核心认知预警：谁来扣动“自动部署”的扳机？

在执行具体操作前，我必须牢记一个底层逻辑：**我的自动化流水线图纸（`deploy.yml`）是存放在主引擎仓库 `my-blog` 里的。**

这就意味着，如果我仅仅在内容仓库（比如 `tech-note`）里推送了代码，云端工厂是不知道需要开工的。每一次更新的最后一步，都必须是“向主引擎发出版控指令”，从而触发整条流水线的运作。

### 场景一：对已有博客进行只字不改的“纯文本修改”

当我在复习时发现以前写的博客有错别字，或者想要补充两句感悟，这是最简单的维护场景，因为不涉及任何新路径的产生。

- **步骤 1：本地修改代码**

  打开本地对应的真实源文件夹（例如 `E:\Web_git\tech-note\`），用 Typora 打开那篇 `.md` 文件，直接修改语句并保存。

- **步骤 2：提交内容仓库**

  打开 GitHub Desktop，左上角切换到 `tech-note` 仓库。填写提交信息（如：`fix: 修正某某文章的错别字`），点击 `Commit` 并 `Push origin`。

- **步骤 3：触发引擎重构（极其重要）**

  此时内容已经上云，但网站还没更新。我需要在 GitHub Desktop 中切换到主引擎仓库 `my-blog`。为了触发流水线，我可以在 `my-blog` 目录下的 `README.md` 文件里随便敲一个空格，或者写上一句更新日志，然后 `Commit` 并 `Push`。

  *(进阶技巧：我也可以在 `deploy.yml` 的 `on:` 下面加一行 `workflow_dispatch:`，这样以后直接去 GitHub 网页端的 Actions 界面，点一下 `Run workflow` 按钮就能手动触发，连假装提交都省了。)*

### 场景二：在已有的知识库中“新建一篇 Markdown 文档”

当我做了一道新的力扣题，或者写了一篇全新的 AI 学习笔记时，由于产生了**新的文件路径**，我不仅要上传内容，还要告诉网站的导航系统“新文章在哪”。

- **步骤 1：本地创作与资源隔离**

  在目标仓库（如 `E:\Web_git\leetcode-solutions\`）中新建 `xx-new-article.md`。使用 Typora 撰写时，牢记将图片保存在自动生成的 `./assets/` 相对路径文件夹中。

- **步骤 2：更新 VitePress 路由字典**

  打开主引擎仓库中的配置文件：`E:\Web_git\my-blog\docs\.vitepress\config.mjs`。找到对应专栏的 `sidebar`（侧边栏）数组，在里面新增一行代码，把这篇新文章的相对路径登记上去。

- **步骤 3：双轨推送（一次性触发）**

  1. 打开 GitHub Desktop，先切换到内容仓库 `leetcode-solutions`，把新文章 `Push` 上去。
  2. 接着切换到主引擎仓库 `my-blog`，把刚才修改过路由的 `config.mjs` 文件 `Push` 上去。

  > 此时，因为 `my-blog` 发生了真正的代码变动，云端流水线会瞬间点火。它会拉取刚刚上传的新文章，并结合新的路由生成完美的静态网页，几分钟后刷新浏览器即可看到新博客上线。

### 场景三：开辟全新领域，将“新仓库”接入个人主页

随着我的技能树不断扩展，我可能会在本地新建一个完全独立的仓库（比如 `E:\Web_git\java-backend-notes`）。要将这个全新的外挂模块接入我的主页大本营，需要进行一次“架构级”的拼装。

- **步骤 1：创建本地虚拟映射（为了本地开发预览）**

  以管理员身份运行 CMD，敲入软链接指令，在本地 `my-blog/docs` 里开一个任意门：

  DOS

  ```
  mklink /J "E:\Web_git\my-blog\docs\java-notes" "E:\Web_git\java-backend-notes"
  ```

- **步骤 2：切断 Git 的误判（极度重要）**

  打开 `E:\Web_git\my-blog\.gitignore` 文件，在最后加上一行 `docs/java-notes/`，防止引擎仓库把这堆虚拟文件误传上云端。

- **步骤 3：更新云端拼装图纸**

  打开核心自动化脚本 `E:\Web_git\my-blog\.github\workflows\deploy.yml`。在“阶段二：微服务模块拉取”中，复制一段 `Checkout` 步骤并修改：

  YAML

  ```
        - name: 2.5 Checkout Java Notes
          uses: actions/checkout@v4
          with:
            repository: G1234-cod/java-backend-notes  # 你的 GitHub 上的新仓库名
            path: docs/java-notes                     # 云端组装的目标路径
  ```

- **步骤 4：配置全局导航**

  在 `config.mjs` 中，不仅要在 `sidebar` 里配这门课的目录，还要在顶部的 `nav` 数组中，加一个可以直达这个新分类的入口按钮。

- **步骤 5：全量推送上线**

  在 GitHub Desktop 中，确保新仓库 `java-backend-notes` 已经提交推送到云端。然后切换回 `my-blog`，将变动后的 `.gitignore`、`deploy.yml` 和 `config.mjs` 一次性 `Push` 到云端。云端机器在接到新图纸后，会自动跑去拉取这个全新的仓库并完成无缝拼装。

### 场景四：架构级代码调整与页面修改边界（The Decision Matrix）

当我想对网站的样式进行微调、优化流水线逻辑、或者更改 VitePress 的高级配置时，这属于“架构级变动”。由于我的系统是多仓库聚合的，**找对修改的源头比怎么修改更重要**。

**第一步：明确代码的修改位置（修改决策矩阵）**

严禁直接在主仓库的 `docs/public/` 等编译目标目录中修改代码，必须永远在“物理源头”进行修改。我需严格遵循以下对照表：

| **我的修改意图**                               | **我必须去哪里修改源文件？**                            | **底层逻辑说明**                                       |
| ---------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------ |
| **调整博客文章的具体内容（文字、图片）**       | `Individual-Resume` 或 `tech-note` 等对应的**内容仓库** | 业务内容的物理源头在这里，供流水线随时拉取。           |
| **修改博客全局导航栏、侧边栏层级、SEO设置**    | `my-blog/docs/.vitepress/config.mjs`                    | 这是主引擎的全局路由字典与控制大脑。                   |
| **调整简历页面的排版、颜色、自定义按钮**       | `Individual-Resume` 源仓库里的 `index.html`             | 简历不受框架控制，局部 UI 组件必须在原生 HTML 中手写。 |
| **更改自动化部署逻辑（如加新仓库、换服务器）** | `my-blog/.github/workflows/deploy.yml`                  | 这是定义云端服务器“怎么干活”的流水线图纸。             |

**第二步：本地沙箱测试与双轨同步**

1. **本地预览：** 在进行任何 HTML 样式或 `config.mjs` 修改后，先在本地通过 `npm run docs:dev` 启动服务，确保排版没有错乱、路由没有断链。
2. **规范提交：** 确认无误后，使用语义化前缀提交代码（如 `style: 优化简历按钮颜色` 或 `refactor: 重构导航栏路由`）。
3. **触发引擎：** 如果修改的是底层子仓库（如简历的 HTML），提交 Push 后，**必须回到主引擎仓库 `my-blog` 触发一次无变动的 Push**（或手动执行 Workflow），让云端流水线重新拉取最新的 HTML 进行覆盖打包。

**第三步：线上回归验证 (Post-Deploy Audit)**

1. **盯盘流水线：** 观察 GitHub Actions 页面，确保 YAML 语法没有引发执行报错。
2. **强制穿透缓存：** 部署绿灯后，打开网站，使用 `Ctrl + F5` 强制刷新，或在开发者工具中开启 `Disable cache`，防止浏览器加载旧的 CSS 缓存导致“修改未生效”的假象。
